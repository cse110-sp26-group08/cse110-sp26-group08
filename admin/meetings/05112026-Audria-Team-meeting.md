## Discussion

### Project Progress Overview

TA Audria noted that the team is currently doing very well overall and appears highly organized compared to other teams in the cohort. Team members were praised for taking the project seriously and maintaining strong collaboration throughout the sprint process.

### Announcements and Upcoming Deliverables

The team discussed several upcoming project requirements and deadlines:

- A public demo is expected by next Thursday
- Teams must prepare a video presentation demonstrating project progress
- Teams will conduct a review of another team’s project that weekend
- The review focus will primarily be on code quality and project understanding rather than feature building

FAudria emphasized the importance of:
- Maintaining a high-quality README
- Keeping ADR documentation updated
- Ensuring that outside reviewers can clearly understand the project structure and goals

### Tester App and Traffic Generation

The team discussed strategies for generating traffic and testing WatchTower’s monitoring capabilities.

The team discussed the need to generate traffic for the tester app so WatchTower can collect measurements and monitor activity. Audria suggested asking friends and other users to visit and interact with the tester app in order to create traffic and provide WatchTower with data to analyze. The team also discussed potentially creating a more engaging or “vibe-coded” site to attract additional users and interactions.

Himir proposed approach involved creating a simple tester application containing:
- Buttons that intentionally trigger frontend errors
- Slow-loading pages
- Pages or actions that intentionally crash the app

The purpose of this tester app would be to allow WatchTower to detect failures and diagnose issues in a controlled environment rather than relying solely on organic traffic.

### MVP and User Stories

Discussion continued around MVP organization and user story planning.

Key points included:
- User stories already exist in the MVP document
- Issues will eventually be created for each user story with more detailed implementation information
- MVP features are prioritized from highest importance to lowest importance on the document
- Additional MVP/stretch features may only be implemented if time permits

### Wireframe Overview

A major portion of the meeting focused on wireframes and UI planning.

Audria described the wireframe as:
- A blueprint for how the frontend will look
- A communication tool between frontend and backend developers
- A way to visualize component responsibilities and application flow

The team asked whether AI-generated UI mockups were acceptable as wireframes, and Audria confirmed that they were acceptable for the current stage of development.

Additional guidance included:
- Wireframes are expected to evolve over time
- Aesthetics are less important than functionality and structure
- Placeholder images and rough layouts are acceptable
- Focus should be placed on page organization and feature allocation

Examples shown during the meeting illustrated:
- User flow diagrams
- Placeholder image layouts
- Page structure planning
- Frontend/backend interaction mapping

### Frontend and Backend Coordination

The discussion also covered how frontend and backend teams should coordinate during development.

Audria emphasized:
- Starting with HTML structure first
- Ensuring all teammates understand page organization
- Treating CSS as lower priority during early implementation
- Making JavaScript functionality and element naming conventions clear to all contributors

How a wireframe can be beneficial to the Backend team:
It clarifies the:
- Element IDs
- Class names
- Expected functionality
- Component interactions
- etc...

### Recommended Tools and Workflow

Several tools for creating wireframes and organizing project ideas were recommended, including:
- Google Slides
- draw.io
- Lucidchart
- Canvas whiteboard tools

The team also discussed how frontend ideas should initially focus on structure and workflow rather than visual polish.
