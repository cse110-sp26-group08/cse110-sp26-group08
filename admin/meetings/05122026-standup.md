## Discussion

### Backend Team Updates

The backend team reported that most MVP issues had already been created and that the basic Express application was now available on the `main` branch.

The team explained the current backend setup process:

- Go into the backend folder
- Run `npm install`
- Install all required dependencies
- Run `npm start` to launch the app

The immediate backend plan was to finish the basic backend tables by that night or the following morning. These were described as initial empty tables that would support later work. After that, the team planned to add metrics-related endpoints, including a collection route such as `/api/collect`.

The next major backend priority after metrics was the login system, which would depend on the frontend having a login page ready.

Other backend work mentioned included:

- Collection endpoints
- Retrieval endpoints
- Page-related endpoints

### Frontend Team Updates

The frontend team reported that it had not yet finished all MVP issues, but expected to finish creating the remaining issues by that night.

In terms of implementation progress, the team said it had:

- Finished the skeletons for most of the planned web pages
- Not yet completed the signup page
- Finished styling for the homepage and login page
- Continued working on the wireframe

The frontend team set two short-term goals for the next standup:

- Finish the wireframe
- Get connected with the backend

The team also discussed the need to update the current frontend link behavior. Right now, the HTML was linking directly to files, but this would eventually need to be changed to use JavaScript routing or URL-based navigation more appropriate for the integrated app flow.

Additional frontend priorities included:

- Styling the dashboard
- Setting up Chart.js
- Using hardcoded data temporarily for charting if needed
- Refining the wireframe so page navigation and button flow are clearly represented

### Collector Team Updates

The collector team said it had reviewed the collector plan in the group repository against the data requirements and previously discussed metrics. Based on that comparison, the team concluded it should focus primarily on client-side data.

The collector priorities identified were:

- Runtime errors
- Performance data
- Client-side feedback and related metrics

The short-term goal before the next standup was to implement the basic collector functions, including:

- Initialize
- Send event
- Track error

The collector team also identified a roadblock: it still needed clarification from the backend team on the exact event payload format expected by the API.

### API and Endpoint Coordination

The team discussed how the collector and backend should coordinate on endpoints and payloads.

The backend team said that the collection endpoints would likely be fixed routes such as:

- `/api/collect/error`
- `/api/collect/performance`

The idea was that when an error occurs, the collector should immediately send that data to the appropriate backend endpoint rather than waiting on a timer. The backend team suggested that fixed endpoints would make implementation easier, since the collector team could hardcode them directly into the file and avoid uncertainty.

The discussion concluded that:

- The collector team should communicate what input shape it needs
- The backend team should define the endpoint structure clearly
- Fixed endpoints are acceptable for now
- Hardcoding endpoint paths is fine at this stage if it helps progress

### Near-Term Team Plan

The meeting ended with a rough division of short-term next steps.

Backend planned to continue database work later that night, with some members coordinating around availability after 9:00 or 10:00 PM.

Frontend planned to:

- Finish dashboard styling

The wireframe was described as something that should show page-to-page transitions clearly, with arrows indicating what happens when users click different buttons.

It was also suggested that within the collector-focused group, one person could build the tester app while the remaining teammates focus on collector implementation.

This standup established a clear short-term plan across the team and clarified how the frontend, backend, and collector work should connect.

1. **Backend setup is now available on `main`:** teammates can install dependencies and start the Express app locally
2. **Most MVP issues have been created:** backend has largely finished issue setup and frontend expected to finish its remaining issue creation that night
3. **Frontend page structure is mostly in place:** homepage and login styling are done, while dashboard work and signup remain
4. **Collector work is now more focused:** the team narrowed its immediate scope to initialization, event sending, and error tracking
5. **Endpoint clarity is the main integration dependency:** the collector team needs backend confirmation on payload shape and route structure
6. **Fixed collection routes are acceptable for now:** hardcoded endpoints such as `/api/collect/error` and `/api/collect/performance` should help the team move faster
7. **The next standup goals are clear:** wireframe completion, backend connection, and continued metrics infrastructure work are the main priorities
