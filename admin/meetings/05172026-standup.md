## Discussion

### Backend Team Updates

The backend team reported substantial progress this sprint across infrastructure, authentication, API design, validation, and collector-related backend work.

Contributions mentioned included:

- Setting up the MongoDB schema, including fields and controllers
- Reviewing pull requests
- Adding input validation and type-safety protections to data manipulation functions
- Researching future sprint work, including false alarms and prototype planning
- Initializing collector endpoints
- Building the login and signup feature
- Creating REST API endpoints for the backend

The backend team generally felt that it completed the sprint goals it originally set, with some members stating the team could take on more work next sprint.

### Frontend Team Updates

The frontend team reported that it completed the major page work planned for the sprint and made strong progress on dashboard structure and wireframe planning.

Contributions mentioned included:

- Completing the homepage, login page, and dashboard pages
- Building the HTML outline for the dashboard
- Adding assets
- Researching the frontend ADR
- Working on the wireframe and feature planning
- Researching how to connect `collector.js` with the dashboard
- Finalizing dashboard work that was still in progress

The frontend team described itself as mostly on target, with most planned work completed and some remaining effort focused on integration and final dashboard polish.

### UX and UI Team Updates

The UX and UI team reported that the collector script reached a functional baseline during this sprint.

Contributions mentioned included:

- Researching the APIs needed for `collector.js`
- Finding minor bugs in the backend server and collector while building the tester app
- Creating the skeleton of the collector script
- Implementing error tracking
- Working on `TrackPerformance`
- Working on `TrackAPIPerformance`

The team said its main goal was to have a somewhat working collector script by the end of the sprint, and members agreed that this goal was largely achieved.

### Sprint Progress Estimates

The thread included rough estimates from each contributor about how much planned work their team completed relative to sprint expectations.

Reported estimates and themes included:

- Backend contributors generally felt they met or nearly met sprint expectations
- Frontend contributors said they completed most or all planned work, with one estimate around 95%
- UX and UI contributors estimated roughly 90% to 100% completion depending on how strictly uptime and availability tracking were counted

Overall, the team’s self-assessment suggested strong progress this sprint, with most subteams believing they completed the core goals they originally committed to.

### Remaining Work

Although progress was strong, several follow-up tasks were identified:

- Backend still needs work on developer notifications for error or downtime events
- False alarm research and prototype work still need to be completed
- Work remains on ensuring the website and collector do not slow down the monitored app
- Frontend still needs to connect more fully with the backend and collector script
- Dashboard finalization and polish are still in progress
- UX and UI still need to implement uptime and availability tracking
- Some collector functions may expand depending on frontend and backend needs

Most contributors believed their remaining sprint work could be finished by the end of Sunday. The main exception was uptime and availability tracking, which several contributors said would likely extend beyond Sunday because it still required research and implementation work.

### Completed MVP User Stories

The thread identified multiple completed MVP user stories, along with additional foundational backend work that supported future stories.

Reported completed user stories included:

- As a developer, I want not make much changes to my existing code and want to just add a single collector file to my repo to collect performance/error data, so that my code stays clean and adding WatchTower is not a cumbersome process.
- As a developer, I want to be able to send errors, performance metrics, availability data to the WatchTower APIs so that I can see them in the visual dashboards which it provides.
- As a developer who values security, I want my data to be protected by my email and encrypted password through a login system, so that I can be worry free using WatchTower in my apps.

It was also noted that a significant amount of backend time went into initialization and database setup work that did not map directly to a named MVP user story, but was still necessary for the product.

## Takeaways and Summary

This standup showed that Sprint 2 made substantial progress across all three subteams and that most of the originally planned work was completed.

1. **Backend made strong implementation progress:** database schema, authentication, endpoints, validation, and collector backend support were all advanced
2. **Frontend completed major page work:** homepage, login, dashboard structure, and wireframe progress were all reported
3. **UX and UI delivered a functional collector baseline:** error tracking and performance-tracking foundations are now in place
4. **Most teams met sprint expectations:** self-reported completion estimates were generally high across the board
5. **Integration work is now the main next step:** dashboard, backend, and collector coordination still need to be tightened
6. **Uptime and availability tracking remain the biggest unfinished gap:** several contributors said this still needs research and may not be finished by Sunday
7. **Multiple MVP user stories were completed:** especially around authentication and lightweight collector integration

**Key follow-up items:**

- Finish dashboard integration with the backend and collector
- Complete remaining dashboard polish
- Continue work on notifications and false alarm handling
- Research and implement uptime and availability tracking
- Improve performance safeguards so WatchTower does not slow the monitored app
- Carry remaining integration work into the next sprint if needed
