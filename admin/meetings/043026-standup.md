## Standup Contributions

### Socio-Technical Nature of Software Engineering

**Contributor:** Yusuf Damda

Software Engineering is fundamentally a socio-technical process, not just source code. Watchtower exemplifies this principle—its core purpose is to understand how systems behave in real-world conditions through user feedback, performance monitoring, and deployment tracking. The system captures how external factors (user feedback, system monitoring) directly influence software development decisions.

**Application to project:** We must remember that our software exists within a larger ecosystem. Success depends not just on writing code, but on understanding user needs and system behavior.

### Mobbing for Code Standards and Team Cohesion

**Contributor:** Ki Diaz

Mobbing (collaborative group coding) is particularly valuable early in projects for establishing code standards and team agreement. Early mobbing sessions can establish:

- Common naming schemes for variables
- Variable declaration conventions
- Function organization patterns
- Commenting and formatting standards
- Commit message formats

Benefits include standardized code quality and improved team cohesion.

**Application to project:** We should consider mobbing sessions early in sprint 2 to establish coding standards before heavy development begins.

### Rough Wireframes Over High-Fidelity Designs

**Contributor:** Yang Bie

During the design phase, rough wireframes should come before high-fidelity designs. Rough wireframes focus team attention on:

- User flow and journey
- Information structure
- Feedback mechanisms

Creating high-fidelity designs too early wastes time on colors, spacing, and visual details before identifying actual workflow problems.

**Application to project:** In our design sprint (Week 2), we should prioritize rough wireframes showing user flows for the dashboard and data collection interfaces before investing effort in visual polish.

### Non-Linear Iterative Agile Process

**Contributor:** Felix Tong

The iterative Agile process differs fundamentally from linear waterfall development. We have been assigned multiple sprints with built-in review and redesign phases. After each sprint, the team meets to discuss accomplishments, identify what needs redesign, and adjust approach for the next sprint. This contrasts with waterfall where design is finalized before development begins.

**Application to project:** We should embrace the iterative nature—expecting that designs and implementations will evolve across sprints rather than being "perfect" from the start. Each sprint review is an opportunity to improve based on what we've learned.

### Top-Down and Bottom-Up Models

**Contributor:** Vinh Tong

Both top-down and bottom-up approaches serve different project phases:

- **Top-down:** Used in sprint planning and high-level meetings to establish broad goals and vision (e.g., "build a dashboard that shows error trends")
- **Bottom-up:** Used during execution to specify lower-level components and implementation details (e.g., individual functions and API endpoints)

**Application to project:** Sprint planning should use top-down thinking (broad goals), while sprint execution should use bottom-up thinking (component-level tasks assigned to individuals and pairs).

### Team Composition Approaches

**Contributor:** Himir Desai

Two primary team composition models exist:

1. **Specific positions approach:** Team members have defined roles (frontend dev, backend dev, UI designer, test engineer, etc.)
2. **Generic positions approach:** All members are generalist developers with occasional specialized roles

The craft literature suggests specific positions are generally better unless the team is very small. Our current composition uses the generic approach (all are developers with some special roles).

**Application to project:** The team should discuss whether switching to specific positions would be feasible and beneficial for our project. This relates to the earlier decision to create functional teams (Backend/Data, Dashboard/UI, Product/UX, DevOps) which moves us toward the specific positions model.

### Bus Factor and Knowledge Distribution

**Contributor:** Matthew Beaudin

The "bus factor" measures how many team members can be removed from the project before it stalls (i.e., hit by a bus). Knowledge concentration creates risk. In projects with short timelines, it's critical to ensure knowledge is distributed throughout the team so that:

- If someone gets sick, the project doesn't halt
- If someone gets unexpectedly busy, others can continue
- No single person becomes a bottleneck

**Application to project:** Given our compressed timeline, we must actively prevent knowledge silos and ensure critical information is shared across team members.

### Pair Programming Strategy

**Contributor:** Matthew Bozoukov

Building on the bus factor concept, pair programming can mitigate risk by ensuring at least two people understand each part of the codebase. This reduces the bus factor and prevents project stalls if one team member becomes unavailable.

**Application to project:** We should consider pair programming (especially for critical components) as a risk mitigation strategy. This also aligns with the earlier discussion about code reviews and pull requests—pairs can review each other's work and share knowledge.

## Takeaways and Summary

- **Socio-technical perspective:** Remember that Watchtower serves real users with real needs; technical excellence alone isn't sufficient
- **Early standards matter:** Mobbing sessions early in development establish code standards and team alignment
- **Design with rough sketches first:** Start with rough wireframes to validate user flow before investing in visual design
- **Embrace iteration:** Agile sprints include built-in redesign opportunities; first versions don't need to be perfect
- **Mix top-down and bottom-up thinking:** Use top-down for planning, bottom-up for execution
- **Consider team structure:** Evaluate whether moving toward specific roles would improve project efficiency
- **Distribute knowledge:** Actively prevent knowledge silos through pair programming and knowledge sharing
- **Pair programming as risk mitigation:** Ensures redundancy and prevents bottlenecks on critical code

This asynchronous standup successfully connected course concepts to real project challenges while honoring team members' exam preparation needs.
